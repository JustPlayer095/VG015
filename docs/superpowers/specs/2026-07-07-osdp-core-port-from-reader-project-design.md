# Port OSDP core from READER_PROJECT onto K1921VG015

Дата: 2026-07-07

## Контекст

`C:\_WORKSPACE\READER_PROJECT` (AT32F415, Keil) — отдельный, гораздо более взрослый проект с своим OSDP-стеком (`Common/OSDP/`), который уже используется в трёх реальных ридерах (Rdr125, Rdr13m, RdrMulty). Наш `common/osdp/` на K1921VG015 — новее, проще, специфичен под один продукт.

Решение: не добавлять `osdp_FILETRANSFER` в READER_PROJECT отдельно (как обсуждалось раньше), а сначала **портировать их OSDP-ядро на K1921VG015**, заменив нашу текущую реализацию. После валидации на отладочной плате K1921 — отдельным follow-up добавить `osdp_FILETRANSFER` поверх уже общего кода, и фича сразу станет доступна в обоих проектах.

Это Phase A. `osdp_FILETRANSFER` — Phase B, отдельный спек, вне скоупа этого документа.

## Скоуп порта

**Переносим (core + базовые команды):**
- Диспетчер команд (`OsdpCheckCommand`-эквивалент, switch/case стиль вместо нашего `core/dispatch/executor`).
- Команды: `POLL`, `ID`, `CAP`, `LED`, `OUT`, `COMSET`, `ISTAT`, `OSTAT`, `BUZ`.
- `OsdpCommands.h` (коды команд/ответов), `OSDPQueue.{c,h}` (логика сборки команды из байт), `CcittCrc16.{c,h}`.

**НЕ переносим** (reader-специфика, неприменима к K1921):
- Всё, что касается Mifare (`mfg_SETMIFSEC`, `mfg_SETCARDP`, sector/key), AES-крипто (`OsdpCrypto.c`, `osdp_KEYSET`, `mfg_KEY_SET`), `ReaderConfig`/`tCapability`/`tDeviceIdReport` в их текущем виде, keypad-мультиплекс из `Keypad.c`.

**Переписываем как новые case в портированном dispatch** (наша текущая функциональность, которую нельзя терять):
- RAW card (`osdp_RAW`, 0x50)
- keypad (`osdp_KEYPAD`, 0x53) + логика wiegand PIN
- смена PIN mode (`mfg_CHGPINMOD`, 0x27)
- ISTAT/OSTAT по-битный ответ (byte-per-bit, недавно добавили)
- OUT/LED control (наши текущие обработчики)

## Архитектурное решение: RX/TX модель

Это главное отличие, ради которого делается порт архитектуры, а не только команд.

**Наша текущая модель (K1921, `vg015/main.c:46-58`):** весь путь — приём байта, сборка кадра, парсинг, dispatch, хендлер, построение ответа, **и блокирующая отправка** (`osdp_port_send_blocking`, busy-wait на TX FIFO) — происходит **внутри одного прерывания UART RX**. Работает сейчас, но означает: пока обрабатывается команда, все прерывания того же/ниже приоритета (наш 1мс-тик `tmr32_irq_handler`, от которого зависят wiegand-таймауты и LED-блинк) не выполняются.

**Модель READER_PROJECT:** RX-прерывание (`USART1_IRQHandler`) только копит байты в буфер/структуру и выставляет флаг `Ready` — выходит немедленно. Обработка команды (dispatch, хендлер, построение ответа) — в главном цикле (mainline), не в прерывании. Отправка ответа — тоже по прерыванию (TX по TDBE/TDC), не busy-wait.

**Почему это важно именно сейчас:** `osdp_FILETRANSFER` (Phase B) неизбежно приносит `spiflash_erase`/`spiflash_write` внутрь обработки команды — это миллисекунды-десятки миллисекунд. Если оставить нашу модель "всё в IRQ", это заблокирует все прерывания на это время. Решение: взять модель READER_PROJECT — IRQ лёгкий (только байты), обработка в mainline, TX не блокирующий.

**Что это значит для порта:**
- `uart4_irq_handler` на K1921 переделывается: только сборка байт в структуру-аналог `tOsdpCommand`, выставление `Ready`, выход.
- Новый вызов в mainline (главный цикл `main.c` или тик-функция) — обработка готовой команды.
- `osdp_port_send_blocking` заменяется на interrupt-driven отправку (TX по прерыванию UART4, не busy-wait цикл).

**Важно:** регистровый код (`OsdpUart.c` в READER_PROJECT) весь на AT32 USART1 API (`usart_data_transmit`, `gpio_bits_set` и т.д.) — **не портируется дословно**. Портируется только логика/структуры данных (как разбирается кадр, как копится команда), регистровый код под K1921 UART4 (PL011-подобный: `IBRD/FBRD/LCRH/IMSC/ICR`) пишется заново.

## Структура кода

- `common/osdp/` — заменяется портированным ядром (dispatch + команды выше). Текущее содержимое (`core/`, `handlers/`, `executor/`, `policy/`, `state/`) уходит целиком.
- `vg015/modules/osdp_port/` — переписывается под новую RX/TX модель на K1921 UART4 регистрах.
- Текущий `common/osdp/` физически нигде не дублируется — он уже в git-истории (смёржен через PR #4 в `main`). Перед началом замены ставится git-тег на текущий `main` (например `pre-reader-osdp-port`) для быстрого отката/сравнения.

## Валидация

Порт делается на новой ветке. После реализации — тестирование на отладочной плате K1921 (не на реальном ридере). Проверяется: весь текущий функционал (карты, keypad, PIN mode, LED/OUT, ISTAT/OSTAT, cards format menu) работает как раньше, плюс новая RX/TX модель не ломает таймингов (wiegand, LED-блинк) во время обработки OSDP-команд.

## Non-goals этого спека

- `osdp_FILETRANSFER` — отдельный follow-up после валидации Phase A.
- Любые изменения в READER_PROJECT (порт делается только на K1921 стороне; READER_PROJECT останется как есть до Phase B).
- Mifare/AES/крипто-функциональность — не переносится, не нужна на K1921.
- Известные баги baseline READER_PROJECT (сломанный билд Rdr125 — отсутствующие файлы `ReaderEHCP.c`/`i2c.c`, отсутствие define `READER_125`) — вне скоупа, решаются отдельно пользователем.
