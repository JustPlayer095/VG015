## Модуль `osdp`

Этот модуль принимает команды по UART и управляет устройством по протоколу OSDP.

### Что важно знать в первую очередь

- Внешний интерфейс из `osdp.h`:
  - `osdp_init()`
  - `osdp_on_rx_byte(uint8_t byte)`
  - `osdp_tick_1ms()`
- Модуль разбит на небольшие части, чтобы его было проще поддерживать и безопаснее менять.
- Главный принцип: код, который "трогает железо", вынесен в отдельный слой `port`.

### Как это работает по шагам

1. В `main.c` приходит байт из UART и передается в `osdp_on_rx_byte()`.
2. Внутренний парсер собирает полный пакет и проверяет его целостность.
3. Команда отправляется в нужный обработчик (`basic`, `control`, `admin`, `filetransfer`).
4. Обработчик формирует ответ и, если нужно, просит выполнить действие (например, сменить baud или reset).
5. Эти действия выполняются централизованно через `executor` и `port`.

### Что за файлы и зачем они нужны

- `osdp.c` — "входная точка" и маршрутизация (тонкий фасад).
- `osdp_runtime.c` — основная рабочая логика (ответы, LED/OUT, file transfer, состояние).
- `core/` — базовая "механика" протокола:
  - разбор входящих байтов,
  - сборка ответных пакетов,
  - выбор обработчика по коду команды.
- `handlers/` — логика по группам команд.
- `executor/` — применяет важные действия (например reset) в одном месте.
- `port/` — весь низкоуровневый доступ к железу:
  - UART, GPIO, reset, delay,
  - extflash для file transfer,
  - update flag.
- `policy/` — проверка и нормализация настроек адреса и скорости.
- `state/` — единая структура состояния `osdp_context_t`.

### Гарантии поведения

- Пакеты принимаются только:
  - на адрес этого устройства,
  - или на broadcast `0x7F`.
- Прямой доступ к железу из обработчиков не допускается.
